# 实操教程：从真实材料生成专属人味 Skill

## 1. 准备材料

建议至少准备四类：

1. 你自己写过并且满意的文章；
2. 未经整理的口述、聊天或语音转写；
3. AI 改稿前后的对照；
4. 你对改稿结果的真实反馈。

不要只给“成品文章”。成品只能说明你最后接受了什么，改稿反馈才能说明你为什么接受或否定。

## 2. 让 Agent 先读参考项目

让 Agent 阅读：

- [orange2ai/renwei-writing](https://github.com/orange2ai/renwei-writing)
- 本仓库的 `PROMPT.md`

参考项目提供的是“保住作者存在感”的方法，不是让 Agent 模仿参考作者。

## 3. 先生成风格档案，不要直接生成 Skill

让 Agent 先填写 `templates/STYLE_PROFILE.md`。

这一步的目的，是把证据和结论分开：

- 证据来自哪篇文章、哪段对话；
- 哪个习惯出现过几次；
- 哪些只是某次情绪；
- 哪些确实可以写进长期规则。

如果 Agent 只凭一两句话就宣布“你的风格是直接、犀利、专业”，说明它还没有真正分析。

## 4. 再生成 SKILL.md

确认风格档案基本准确后，再让 Agent 按 `templates/SKILL_TEMPLATE.md` 输出 Skill。

一份可用的 Skill 应该包含：

- 作者站在什么位置说话；
- 文章通常怎么起头；
- 观点如何展开；
- 哪些毛边必须保留；
- 哪些词和句式必须避免；
- 修改前如何判断该不该动；
- 修改后如何验收；
- 真实正反案例。

## 5. 做盲测

选择一篇没有参与风格分析的旧文章，让 Agent 改写。

不要告诉它你原来的最终版本。只给原稿和 Skill，让它独立完成，再比较结果。

重点检查：

- 观点有没有被改软；
- 技术判断有没有被抽象化；
- 合理的犹豫有没有被删掉；
- 有没有增加作者不会使用的文学场景；
- 有没有出现整齐得过头的结构；
- 有没有把原本自然的重复当成错误清理掉。

## 6. 把失败原因写回 Skill

不要只在对话里说“这次不像我”。

每次反馈都应该变成规则，例如：

```text
反馈：这段太像行业报告。
规则：产品和技术判断优先使用具体功能、成本和执行路径，不替换成“生态、赋能、场景落地”等抽象词。
```

```text
反馈：你把我的判断改得太客观了。
规则：不要为了平衡而主动补反方观点。保留作者明确的个人立场，只在事实错误时纠正。
```

## 7. 维护版本

建议给 Skill 维护简单版本记录：

```text
v0.1：根据历史文章生成初版
v0.2：补充公众号长文结构规则
v0.3：修复过度删除“我觉得”的问题
v0.4：加入技术类比和产品判断案例
```

Skill 不是一次性提示词，更像一个长期维护的个人表达配置。

## 推荐工作流

```text
准备材料
  ↓
读取参考项目与 PROMPT.md
  ↓
生成 STYLE_PROFILE.md
  ↓
人工快速校正
  ↓
生成 SKILL.md 与检查清单
  ↓
用未见过的旧文章盲测
  ↓
把失败原因写回 Skill
  ↓
形成稳定版本
```

## 判断是否成功

成功不是文章“更流畅”或者“更高级”。

真正的标准是：

> 文章改顺了，但说话的人没有被换掉。
